# 웹툰 등록 자동화 상세 진행 절차 (요구사항/제약/변경관리)

## 1. 비즈니스 골 설정 방법
프로젝트 시작 시 아래 4가지를 먼저 확정합니다.
- 무엇을 얼마나 줄일 것인가: 등록 시간, 재작업 횟수
- 어떤 품질을 확보할 것인가: 누락/오입력 최소화
- 어떤 범위부터 시작할 것인가: 플랫폼/작품/회차 범위
- 어떤 방식으로 성공을 판단할 것인가: KPI와 측정 시점

## 2. As-Is Flow로 현재 상황 확인
As-Is 분석에서 확인하는 항목:
- 회사별 실제 작업 순서(로그인 -> 메타 입력 -> 파일 업로드 -> 제출)
- 반복이 많은 구간과 오류가 자주 나는 구간
- 예외 상황(권한 오류, 업로드 실패, 검증 실패 등)
- 사람이 반드시 판단해야 하는 단계

As-Is 결과를 기반으로, "자동화 후보"와 "수동 유지 구간"을 나눕니다.

## 3. 자동화할 부분 도출 방식
자동화 우선순위는 아래 기준으로 선정합니다.
- 반복 빈도가 높은가
- 입력 오류가 자주 발생하는가
- 규칙으로 판단 가능한가
- 플랫폼별 차이를 설정으로 흡수 가능한가

예시:
- 우선 자동화: 메타 입력 반복, 파일 업로드 반복, 사전 검증
- 반자동/수동 유지: 최종 승인, 예외 케이스 수동 판단

## 4. 요구사항 명세 체계
### 4.1 기능 요구사항 (무엇을 해야 하는가)
| 구분 | 요구사항 |
|---|---|
| FR-01 | 업로드 전 사전 검증(누락/형식/파일 규격) |
| FR-02 | 플랫폼별 등록 자동 실행 |
| FR-03 | 제출/완료 처리 및 상태 저장 |
| FR-04 | 실패 사유 리포트 제공 |
| FR-05 | 재시도 가능한 오류와 수동 전환 오류 구분 |

### 4.2 비기능 요구사항 (어떤 품질로 해야 하는가)
| 구분 | 요구사항 |
|---|---|
| 안정성 | 실패 시 원인과 대응 절차가 명확해야 함 |
| 보안 | 계정/권한 정보 안전 관리 |
| 운영성 | 실무자가 결과를 쉽게 해석하고 조치 가능 |
| 확장성 | 플랫폼 추가 시 구조 변경 최소화 |
| 추적성 | 실행 이력과 변경 이력 확인 가능 |

## 5. 제약사항 정리
| 제약 | 프로젝트에 미치는 영향 |
|---|---|
| 플랫폼 정책/화면 차이 | 플랫폼별 처리 분기 필요 |
| API 제공 여부 불확실 | 웹 자동화 방식 검토 필요 |
| 초기 인력/일정 제한 | MVP 우선순위 중심으로 범위 제한 |

## 6. 변경 가능하도록 해야 하는 요구사항 (중요)
초기 정의 후에도 바뀔 수 있는 항목은 "수정 가능" 구조로 설계합니다.
- 플랫폼별 필수 입력 필드
- 검증 규칙(금칙어, 파일 규격, 필수 항목)
- 승인 정책(어디서 사람 확인이 필요한지)
- 재시도 정책(횟수/대상/수동 전환 기준)

변경 방식:
1. 변경 요청 등록
2. 영향 범위 확인(플랫폼/기능/운영)
3. 고객 승인
4. 문서 업데이트 후 반영

## 7. To-Be 시퀀스(고객이 보는 실행 순서)
```text
[실행]
 -> [사전 검증]
 -> [업로드]
 -> [제출/완료]
 -> [결과 확인]
 -> [실패 건 재처리 또는 수동 처리]
```

## 8. 단계별 승인 게이트
| 시점 | 고객이 승인할 내용 |
|---|---|
| Gate 1 | As-Is가 실제 업무와 일치하는지 |
| Gate 2 | MVP 범위와 자동화/수동 경계가 타당한지 |
| Gate 3 | 요구사항 명세(기능/비기능/제약)가 충분한지 |
| Gate 4 | 시연 결과가 실무 적용 가능한지 |
| Gate 5 | 운영 전환 가능한 품질인지 |

## 9. 고객과의 운영 방식
- 정기 리뷰에서 결정 필요 항목을 우선 공유
- 합의된 내용은 즉시 문서 반영
- 변경사항은 이력으로 관리하여 추적 가능하게 유지

## 10. 이번 단계에서 고객이 확정해주실 항목
- 비즈니스 목표 우선순위(시간/품질/범위)
- 자동화 우선 구간과 수동 유지 구간
- 변경 가능 항목의 승인 절차
- 1차(MVP) 적용 범위
