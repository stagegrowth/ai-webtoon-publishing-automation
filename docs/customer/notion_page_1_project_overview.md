# 웹툰 등록 자동화 프로젝트 진행 방식 (고객 안내)

## 1. 문서 목적
이 문서는 "이 프로젝트가 어떤 절차로 진행되는지"를 고객이 이해할 수 있도록 설명합니다.

## 2. 프로젝트를 진행하는 기본 원칙
- 현재 업무(As-Is)를 먼저 정확히 확인한 뒤 자동화 범위를 정합니다.
- 기능 요구사항과 비기능 요구사항을 분리해 명세합니다.
- 제약사항을 먼저 합의하고, MVP(1차 범위)를 현실적으로 확정합니다.
- 단계별 산출물과 승인 기준을 두고 진행합니다.

## 3. 전체 진행 절차 (상위 단계)
1. 비즈니스 목표 설정
- 시간 단축, 오류 감소, 운영 표준화 등 목표를 수치/기준으로 정의

2. 이해관계자 정리
- 실무 담당자, 의사결정자, 운영 관리자, 개발 담당자의 역할 분리

3. As-Is 확인
- 회사별 실제 등록 절차, 예외 상황, 반복 업무를 문서화

4. 요구사항 도출 및 명세
- 기능 요구사항: 시스템이 반드시 해야 하는 동작
- 비기능 요구사항: 안정성/보안/운영성/확장성 기준

5. 제약사항 합의
- 플랫폼 정책, API 유무, 일정/인력 제약을 반영하여 실행 가능한 범위 확정

6. To-Be 시퀀스 설계
- 자동화할 단계와 사람 확인이 필요한 단계를 분리

7. 구현 및 검증
- 사전검증(dry-run) -> 실제 업로드 -> 제출/결과 확인 순으로 단계 구현

8. 운영 전환
- 시범 운영 후 보완사항 반영, 정식 운영 전환

## 4. Stakeholder와 역할
| 이해관계자 | 주요 역할 | 주요 의사결정 |
|---|---|---|
| 고객 실무 담당자 | 현재 업무 설명, 테스트 참여 | As-Is 정확성, 실무 적용성 |
| 고객 의사결정자 | 우선순위/범위 승인 | MVP 범위, 자동화 경계, 승인 절차 |
| 개발 담당 | 설계/구현/테스트 | 기술 방식, 오류 대응, 운영 안정성 |

상세 이해관계자 목록(역할·관심사·영향력·소통): [design/overview/stakeholder_list.md](../design/overview/stakeholder_list.md)

## 5. 비즈니스 Context Diagram
요약:
```text
[고객 운영팀]
  -> (엑셀/원고 파일, 운영 정책)
[자동화 시스템]
  -> (검증/업로드/제출)
[플랫폼 포털들]
  -> (성공/실패 결과)
[결과 리포트]
  -> (고객 확인/승인/재작업 지시)
```
상세 다이어그램(텍스트·Mermaid·액터 정의·데이터 흐름): [design/overview/business_context_diagram.md](../design/overview/business_context_diagram.md)

## 6. 단계별 주요 산출물
| 단계 | 산출물 |
|---|---|
| 목표/범위 정의 | 비즈니스 골, MVP 범위표 |
| As-Is 분석 | 회사별 Flow, 예외 시나리오, pain point |
| 요구사항 명세 | 기능 요구사항, 비기능 요구사항, 수용 기준 |
| 설계 | [비즈니스 컨텍스트 다이어그램](../design/overview/business_context_diagram.md), To-Be 시퀀스, 데이터/검증 규칙, 오류 대응 정책 |
| 구현/검증 | 테스트 결과, 시연 기록, 개선 항목 |
| 운영 전환 | 운영 가이드, 점검 체크리스트 |

## 7. 고객이 이 페이지에서 확인할 내용
- 단계 순서가 실제 프로젝트 진행에 맞는지
- 의사결정이 필요한 시점이 명확한지
- 단계별 산출물로 충분히 관리 가능한지
