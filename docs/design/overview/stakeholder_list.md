# Stakeholder List (이해관계자 목록)

## 1. 문서 목적
이 문서는 웹툰 등록 자동화 프로젝트와 관련된 **이해관계자(Stakeholder)**를 정리합니다. 역할·관심사·영향력·소통 방식을 명확히 해 프로젝트 진행과 의사결정을 원활히 합니다.

---

## 2. Stakeholder 목록

| ID | 이해관계자 | 구분 | 주요 역할 | 관심사(Interest) | 영향력(Influence) | 주요 의사결정 |
|----|------------|------|-----------|------------------|-------------------|---------------|
| SH-01 | 고객 | 고객 | As-Is 설명·테스트 참여·원고·메타 입력 / 우선순위·범위·예산·일정 승인·Go/No-Go / 운영 정책·재작업 정책·결과 확인(실무·의사결정·운영 역할 포함) | 등록 시간 단축·오류 감소·실무 적용성, 비즈니스 목표·범위 균형, 운영 표준화·예외 대응 | 요구사항·수용 기준, MVP 범위·자동화 경계·승인 절차, 운영·재시도 정책 반영 | As-Is 정확성·테스트 수용, MVP·Gate 승인·변경 요청, 운영 정책·알림·리포트·재작업 절차 |
| SH-02 | 개발 담당 | 수행 | 설계·구현·테스트, 기술 방식 선택, 오류 대응 설계 | 요구사항 구현, 안정성·유지보수성 | 기술 방식, 아키텍처, 오류 대응·재시도 정책 | 기술 방식, 구현 범위, 운영 전환 조건 |
| SH-03 | (선택) 프로젝트 관리/코디네이터 | 수행 또는 고객 | 일정·산출물 조율, 회의 진행, 변경·이슈 추적 | 단계별 진행, 의사결정 시점 명확화 | 회의·리뷰 주기, 산출물 승인 흐름 | 리뷰 주기, Gate 일정, 이슈 에스컬레이션 |

## 3. 구분 정의

| 구분 | 설명 |
|------|------|
| **고객** | 자동화 결과를 사용하고, 범위·정책·승인을 결정하는 측 |
| **수행** | 자동화 시스템을 설계·구현·운영하는 측 |

---

## 4. 의사결정·승인 관계 (요약)

```text
고객(SH-01)    ── 확인/승인/정책 ──▶  As-Is 정확성, 테스트 수용, MVP·Gate 승인, 운영 정책·재작업·알림
개발 담당(SH-02)  ── 제안/구현 ──▶  기술 방식, 구현 범위, 운영 전환 조건
```

### 4.1 플랫폼(등록 대상 회사) 웹 변경 시 추가 요구사항 반영

| 역할 | 담당 | 설명 |
|------|------|------|
| **반영 요청** | **고객(SH-01)** | 대상 플랫폼 웹이 변경되었을 때, 추가·변경 요구사항 반영을 **요청**하는 주체. 실무에서 화면/절차 변경을 인지하고 요청 등록·우선순위 제안. |
| **접수·분석·구현** | 개발 담당(SH-02) | 요청 접수 후 영향 범위 분석, 요구사항 명세 보완, 설계·구현·테스트 수행. 필요 시 고객과 검증·수용 기준 합의. |

(변경 요청 절차는 [notion_page_2](../../customer/notion_page_2_execution_flow_and_requirements.md) §6 변경 가능 요구사항·승인 방식 참고.)

## 6. 고객이 이 페이지에서 확인할 내용
- 본인(조직)이 위 표의 어느 역할에 해당하는지
- 의사결정이 필요한 시점(예: Gate 1~5)에 고객 측에서 누가 승인하는지
- 고객 내부에서 실무·의사결정·운영 역할 분담이 명확한지
- 누락된 이해관계자(예: 플랫폼 대응 담당, 보안·인프라 담당)가 있다면 추가할지
