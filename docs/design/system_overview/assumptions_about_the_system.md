# Assumptions about the System (시스템에 대한 가정)

## 1. 문서 목적
웹툰 등록 자동화 시스템을 설계·구현할 때 **전제로 두는 가정(Assumptions)**을 정리합니다. 가정이 깨질 경우 영향 범위를 검토하고 문서를 갱신합니다.

---

## 2. 가정 목록

### 2.1 환경·실행

| ID | 가정 | 설명 |
|----|------|------|
| A-ENV-01 | 시스템은 고객 또는 수행 측이 관리하는 환경에서 실행된다. | 온프레미스·클라우드·로컬 등 배포 주체·환경은 프로젝트에서 합의한다. |
| A-ENV-02 | 원고·메타 소스(폴더·엑셀 등)에 대한 읽기 접근이 가능하다. | 로컬 또는 공유 스토리지 경로를 시스템이 접근할 수 있다고 가정한다. |
| A-ENV-03 | 플랫폼 포털에 네트워크 접근이 가능하다. | 방화벽·프록시 등으로 차단되지 않는다고 가정한다. (제한 시 예외 처리·설정 필요.) |

---

### 2.2 플랫폼·연동

| ID | 가정 | 설명 |
|----|------|------|
| A-PLAT-01 | 대상 플랫폼은 웹 브라우저로 접근 가능한 포털 형태이다. | API 미제공 시 브라우저 자동화(웹 UI)로 등록 흐름을 수행할 수 있다고 가정한다. |
| A-PLAT-02 | 플랫폼별 로그인·입력·업로드·제출 흐름이 자동화 가능한 수준으로 안정적이다. | 화면 구조·URL이 급격히 자주 바뀌지 않는다고 가정한다. 변경 시 유지보수 필요. |
| A-PLAT-03 | 플랫폼 정책상 자동화(봇·스크립트)가 명시적으로 금지되지 않는다. | 이용약관·정책 확인은 프로젝트·고객 책임으로 전제한다. |

---

### 2.3 데이터·입력

| ID | 가정 | 설명 |
|----|------|------|
| A-DATA-01 | 원고·메타의 형식(폴더 구조·엑셀 스키마 등)이 사전에 정의·합의된다. | 시스템은 그 형식에 맞춰 읽고 검증한다. 형식 변경 시 설정·스키마 갱신이 필요하다. |
| A-DATA-02 | 메타데이터는 시스템이 파싱 가능한 형태(엑셀·CSV·JSON 등)로 제공된다. | 비정형 문서만 있는 경우 파싱 전처리는 별도 범위로 본다. |
| A-DATA-03 | 원고 파일 규격(확장자·크기·해상도 등)은 플랫폼 요구사항과 합의된 범위 내로 가정한다. | 플랫폼별 상이 시 설정·검증 규칙으로 반영한다. |

---

### 2.4 사용자·운영

| ID | 가정 | 설명 |
|----|------|------|
| A-OPS-01 | 고객 운영팀은 실행 대상(작품·회차·플랫폼)과 소스 위치를 지정할 수 있다. | UI·CLI·설정 파일 등 구체적 방식은 설계에서 정한다. |
| A-OPS-02 | 실패·예외에 대한 대응은 리포트·알림을 보고 사람이 판단·조치하는 것을 전제로 한다. | 완전 무인 자동 복구는 범위 외로 가정할 수 있다. |
| A-OPS-03 | 개발/운영 담당이 설정·배포·모니터링을 수행할 수 있는 권한과 도구를 갖는다. | 계정·배포 파이프라인 등은 프로젝트·고객 환경에 따른다. |

---

### 2.5 보안·계정

| ID | 가정 | 설명 |
|----|------|------|
| A-SEC-01 | 플랫폼 로그인에 사용하는 계정·비밀번호 또는 API 키는 안전하게 저장·주입되는 방식을 쓴다. | 저장소·암호화 방식은 보안 요구사항에 따라 설계한다. |
| A-SEC-02 | 자동화용 계정(서비스 계정 등) 사용이 플랫폼·고객 정책상 허용된다고 가정한다. | 정책 확인은 고객·프로젝트 책임으로 전제한다. |
| A-SEC-03 | 플랫폼 로그인 비밀번호(및 API 키 값)는 **고객 운영팀**이 설정(또는 설정 UI)에서 직접 갱신할 수 있다. | 비밀번호 값만 변경되는 경우 개발자 수정·배포 없이 사용자(고객 운영팀)가 대응할 수 있다고 전제한다. |

---

### 2.6 변경·유지보수

| ID | 가정 | 설명 |
|----|------|------|
| A-CHG-01 | 플랫폼 웹/API 변경 시 요구사항 반영·코드 수정이 발생할 수 있다. | 변경 감지·대응 절차는 별도로 정한다. (고객이 반영 요청 주체.) **인터페이스(웹 UI) 변경 시 시스템 수정은 개발팀이 수행하는 것으로 정의한다.** |
| A-CHG-02 | 검증 규칙·재시도 정책·필수 필드 등은 설정으로 변경 가능한 구조로 둔다. | 코드 수정 없이 설정만으로 조정 가능하다고 가정한다. |
| A-CHG-03 | **요소 ID(셀렉터) 맵핑**은 고객 운영팀이 **사용자용 UI**에서 직접 등록·수정할 수 있다. 플랫폼 웹 요소 ID가 변경되어도 고객이 맵핑만 수정하여 시스템을 복구·유지할 수 있다. | 맵핑 데이터 구조·UI 요구사항은 [element_id_mapping.md](element_id_mapping.md) 정의. |

---

## 3. 가정 검증·갱신
- 프로젝트 초기 또는 플랫폼 추가 시 위 가정이 실제와 맞는지 확인한다.
- 가정이 깨진 경우(예: API 금지, 형식 변경) 영향 범위를 분석하고, 설계·요구사항·이 문서를 갱신한다.
- 새로운 가정이 생기면 이 목록에 ID와 함께 추가한다.
