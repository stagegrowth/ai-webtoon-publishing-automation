# Quality Attributes (품질 속성)

## 1. 문서 목적
아키텍처 결정에 영향을 주는 **품질 속성(Quality Attributes)**을 정리합니다. 이 시스템에서는 **변경용이성(Modifiability)**을 최우선으로 두고, 성능은 상대적으로 부차적으로 둡니다.

---

## 2. 품질 속성 우선순위

| 우선순위 | 품질 속성 | 비고 |
|----------|-----------|------|
| **1** | **변경용이성 (Modifiability)** | 플랫폼 웹 UI 변경·로그인 방식 변경·플랫폼 추가에 대응 가능해야 함. 아래 §3에서 상세 기술. |
| 2 | 유지보수성·운영성 | 설정·로깅·리포트로 실무자가 조치 가능. |
| 3 | 안정성·추적성 | 실패 시 원인·대응 명확, 실행 이력 확인 가능. |
| — | 성능 | 대량 동시 등록 등 극단적 부하는 1차 범위 외. 일반적인 회차·플랫폼 단위 실행에서 큰 이슈 없음. |

---

## 3. 변경용이성 (Modifiability)

대상 플랫폼의 웹 인터페이스 변경, 로그인 방식 변경, 플랫폼 추가가 발생해도 **핵심 로직 변경을 최소화**하고 **플랫폼별 수정·추가만으로 대응**할 수 있도록 설계합니다.

### 3.1 플랫폼 웹 인터페이스 변경 대응

| 구분 | 내용 |
|------|------|
| **시나리오** | 대상 플랫폼(카카오·네이버·레진·교보 등)의 웹 UI(URL·폼·버튼·DOM 구조)가 변경된다. |
| **기대 반응** | 해당 플랫폼에 대한 **플랫폼별 어댑터(또는 설정)**만 수정·배포하여 대응할 수 있다. 공통 흐름(검증·실행 오케스트레이션·리포트)은 변경하지 않는다. |
| **아키텍처 시사** | 플랫폼별 연동 로직을 **공통 인터페이스 뒤로 분리**하고, 플랫폼 단위로 구현체(웹 자동화 스크립트·셀렉터·플로우)를 두어, 한 플랫폼 변경 시 해당 구현체만 수정·교체 가능하게 한다. URL·셀렉터·단계 순서 등은 설정으로 두어 코드 수정 없이 일부 조정 가능하면 좋다. |

### 3.2 로그인 방식 변경 대응

| 구분 | 내용 |
|------|------|
| **시나리오** | 대상 플랫폼의 로그인 방식이 바뀐다. (예: ID/비밀번호 → OAuth, 2단계 인증 추가, 캡차 도입, 세션 정책 변경 등) |
| **기대 반응** | 해당 플랫폼의 **인증 전략만** 추가·변경하여 대응할 수 있다. 등록 파이프라인(메타 입력·업로드·제출)은 그대로 두고, “로그인 세션 확보” 부분만 플랫폼별·방식별로 교체 가능하게 한다. |
| **아키텍처 시사** | 로그인·세션 관리를 **플랫폼별 인증 모듈(또는 전략)**로 분리한다. ID/PW, OAuth, API 키 등 방식이 달라도 공통 인터페이스(예: “유효한 세션/토큰 확보”)를 만족하는 구현을 추가·교체할 수 있게 한다. 계정·비밀번호·토큰 등 자격 정보는 설정·보안 저장소에서 주입받도록 하여 코드에 묶이지 않게 한다. |

### 3.3 플랫폼 추가 대응

| 구분 | 내용 |
|------|------|
| **시나리오** | 새로운 대상 플랫폼(웹툰 등록 포털)이 추가된다. |
| **기대 반응** | **기존 플랫폼 코드를 거의 건드리지 않고** 새 플랫폼용 연동·인증 구현을 추가하고 설정만 등록하면 동작한다. 공통 검증·실행 오케스트레이션·리포트·재시도 정책은 재사용한다. |
| **아키텍처 시사** | 플랫폼을 **플러그인/어댑터 단위**로 추가할 수 있는 구조를 둔다. 공통 계약(인터페이스): 로그인·메타 입력·업로드·제출·결과 파싱 등 단계별 연동 시그니처를 정의하고, 신규 플랫폼은 그 계약을 구현한 모듈만 추가한다. 플랫폼 목록·URL·필수 필드 매핑 등은 설정(또는 설정 UI)으로 관리하여 코드 배포 없이 등록 가능하면 좋다. |

## 4. 변경용이성 요약 (체크리스트)

설계·구현 시 아래를 만족하도록 합니다.

| # | 항목 |
|---|------|
| 1 | 플랫폼별 연동 로직이 공통 인터페이스 뒤에 분리되어 있으며, 한 플랫폼 웹 UI 변경 시 해당 플랫폼 구현체만 수정 가능한가? |
| 2 | 로그인·세션 확보가 플랫폼별·방식별 인증 모듈로 분리되어 있으며, 비밀번호·OAuth 등 방식 변경 시 해당 인증 구현만 추가·교체 가능한가? |
| 3 | 신규 플랫폼 추가 시 기존 플랫폼 코드 변경 없이 새 어댑터(및 설정) 추가만으로 등록 가능한가? |
| 4 | URL·셀렉터·필수 필드 매핑 등 플랫폼별 변동 가능 항목은 설정으로 두어, 가능한 범위에서 코드 수정 없이 조정 가능한가? |

---

## 5. 그 외 품질 속성 (요약)

| 품질 속성 | 기대 수준 | 비고 |
|-----------|-----------|------|
| **유지보수성** | 검증 규칙·재시도 정책·필수 필드 등이 설정으로 관리되고, 코드 변경 없이 조정 가능한 범위를 넓힌다. | Assumptions, System Feature List와 연계 |
| **안정성** | 일시적 오류는 재시도, 영구적 오류는 수동 전환 구분·리포트로 원인과 대응이 명확하다. | |
| **추적성** | 실행 이력·상태·에러가 저장·조회·리포트되며, 변경 이력은 필요 시 별도 관리한다. | |
| **성능** | 회차·플랫폼 단위 실행에서 체감 지연이 크지 않은 수준. 대량 동시 등록·초고속 처리 요구는 1차 범위 외. | |
